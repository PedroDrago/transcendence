# Posts Service

Microservice responsible for posts, stories, comments, and likes in the Transcendence platform. Built with [Elysia](https://elysiajs.com/) on [Bun](https://bun.sh/), Drizzle ORM, Redis, and MinIO/R2.

## Architecture overview

```
Frontend (Next.js)
    └── API Gateway (NestJS :4000)
            ├── Posts Service (Elysia :3333)  ← this service
            └── Auth Service (NestJS :4001)
```

Both backend services share a single PostgreSQL instance but use separate schemas (`posts.*` for this service).

## Stack

| Layer | Technology |
|-------|-----------|
| Runtime | Bun 1.3 |
| Framework | Elysia 1.4 |
| ORM | Drizzle ORM + drizzle-zod |
| Database | PostgreSQL 17 (`posts` schema) |
| Cache | Redis 8 |
| Object storage | MinIO (local) / Cloudflare R2 (production) |
| Validation | Zod 4 |
| Observability | OpenTelemetry → Jaeger |
| Linting | Biome via ultracite |

## Getting started

### Prerequisites

- [Bun](https://bun.sh/) >= 1.3
- Docker + Docker Compose

### 1. Start local infrastructure

```bash
docker compose up -d
```

This starts PostgreSQL, Redis, MinIO, and Jaeger.

### 2. Configure environment

Copy the test env as a starting point:

```bash
cp .env.test .env
```

Adjust any values as needed (ports, credentials).

### 3. Run migrations

```bash
bun run db:migrate
```

### 4. Start the dev server

```bash
bun run dev
```

The API is available at `http://localhost:3333`.  
Interactive docs (Swagger UI) at `http://localhost:3333/openapi`.

## Running tests

Tests require the local infrastructure to be running.

```bash
# Run all tests
bun test

# Run a single file
bun test src/http/routes/posts/delete-post.spec.ts
```

Tests use `.env.test` automatically. Each run truncates all tables, flushes Redis, and clears MinIO, so they are safe to run repeatedly.

## API overview

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/presign-url` | Get a presigned upload URL for R2/MinIO |
| `POST` | `/posts` | Create a post |
| `GET` | `/posts` | List posts for a user |
| `GET` | `/posts/:id` | Get a post |
| `PATCH` | `/posts/:id` | Update a post |
| `DELETE` | `/posts/:id` | Delete a post |
| `POST` | `/stories` | Create a story |
| `GET` | `/stories` | List stories for a user |
| `GET` | `/stories/:id` | Get a story |
| `DELETE` | `/stories/:id` | Delete a story |
| `POST` | `/comments` | Create a comment or reply |
| `GET` | `/comments/:id` | Get a comment |
| `GET` | `/posts/:id/comments` | List top-level comments for a post |
| `GET` | `/stories/:id/comments` | List top-level comments for a story |
| `GET` | `/comments/:id/replies` | List replies for a comment thread |
| `PATCH` | `/comments/:id` | Update a comment |
| `DELETE` | `/comments/:id` | Delete a comment |
| `POST` | `/likes` | Create a like (post, story, or comment) |
| `DELETE` | `/likes/:id` | Delete a like |

All routes require a `Authorization: Bearer <jwt>` header. The JWT is verified against a shared secret with the Auth Service.

## Technical decisions

### Elysia plugin composition

The app is assembled by composing Elysia instances via `.use()`. Each route is its own Elysia instance exported from its file; there is no router class or controller layer. Business logic and DB queries live directly in route handlers — no unnecessary indirection for a service of this size.

### Auth macro

Rather than middleware that runs on every request, the auth plugin exposes a `{ auth: true }` option on individual routes. When declared, the macro extracts the Bearer JWT, verifies it, and injects `userId` into the handler context. This keeps the binding explicit and co-located with the route definition.

### Media upload flow

Uploads do not go through this service. The client:
1. Calls `POST /presign-url` to get a presigned URL pointing to a `tmp/<context>/` prefix in R2/MinIO.
2. PUTs the file directly to that URL.
3. Calls `POST /posts` or `POST /stories` with the resulting key.
4. The service moves the file from `tmp/` to the permanent prefix.

This avoids streaming large files through the API server.

### Caching strategy

Individual resources (post, story) are cached in Redis for 600 s. User list pages are cached only for the first page (cursor = null). Signed media URLs have their own cache entry (3000 s TTL). Cache is invalidated by key on any mutating operation — no cache-aside patterns with stale reads.

### Comments threading model

Three nullable FK columns express position:
- `postId` / `storyId` — the top-level target
- `replyId` — the direct parent comment
- `rootId` — the thread root (first-level comment)

`rootId` is `parent.rootId ?? parent.id` when replying, making it trivial to fetch an entire thread with a single `WHERE rootId = ?` query.

### Likes polymorphism

A single `likes` table covers posts, stories, and comments via nullable FK columns. Uniqueness is enforced with partial indexes per target type, so the database rejects duplicates at the constraint level (→ 409 on conflict).

### Ownership checks

Delete and update routes enforce ownership by including `userId` in the `WHERE` clause of the DB query. If the record exists but belongs to another user, the query returns 0 rows → 404. This avoids a separate permission-check round-trip and leaks no information about record existence.

### Drizzle + drizzle-zod

Table definitions are the single source of truth. `drizzle-zod` derives Zod schemas (`createSelectSchema`, `createInsertSchema`, `createUpdateSchema`) directly from the Drizzle table, keeping validation in sync with the DB schema without duplication.

### Observability

`openTelemetryPlugin` is the first plugin applied so it instruments all downstream plugins and routes. Traces are exported via OTLP/HTTP to Jaeger (configurable via `OTEL_EXPORTER_URL`).
